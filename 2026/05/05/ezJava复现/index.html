EzJava 反序列化漏洞分析与利用

在此说明一下，因为是复现，我也不清楚比赛时用的java1.8的具体版本，所以我就直接用我系统上的java版本了

一、环境信息

项目 版本/说明
JDK Java 1.8.0_202
框架 Spring Boot 2.7.18 (Spring Framework 5.3.31)
关键依赖 commons-collections:3.2.1
内置容器 Tomcat 9.0.83
运行方式 java -jar ezjava-1.0.0.jar

二、应用结构

2.1 项目入口 — EzJavaApplication.java

标准的 Spring Boot 应用，无额外配置。

2.2 漏洞端点 — HelloController.java

POST / 端点接收任意二进制数据，直接进行 Java 反序列化。这是 Java Web 应用中极其危险的操作，攻击者可以发送精心构造的恶意对象链来实现远程代码执行（RCE）。

2.3 缓解措施 A：SafeObjectInputStream.java（黑名单过滤器）

作用：仅在反序列化解析类描述符时检查类名前缀。

致命缺陷：

  • resolveClass() 只在 ObjectInputStream 读取序列化流中的类描述符时被调用。
  • 在运行时通过 Class.forName() 动态获取的类**不经过 resolveClass()**。
  • 黑名单只有 6 个条目，commons-collections 的 gadget 类全都不在其中。

2.4 缓解措施 B：CustomSecurityManager.java（网络出口限制）

作用：禁止非回环（127.0.0.1）的网络连接。

效果：限制外连反弹 shell，但不影响：

  • 本地命令执行（写文件、启动进程）
  • 回环地址（127.0.0.1）的网络通信

2.5 依赖 — commons-collections:3.2.1

pom.xml 中显式引入了 commons-collections:3.2.1。该版本是 Java 反序列化历史上最著名的 “gadget” 库，提供了 InvokerTransformer、ChainedTransformer、LazyMap 等类，可以通过反射链执行任意代码。经典攻击工具 ysoserial 最初就是基于此库开发的。

3.3 黑名单绕过原理

黑名单检查 resolveClass() 的时机：

关键点：

  1. 链中用 ConstantTransformer(Class.class) 取代 ConstantTransformer(Runtime.class)
  2. Class.class 不在黑名单，顺利通过序列化
  3. Runtime.class 从未出现在序列化流中
  4. 只在运行时通过 Class.forName("java.lang.Runtime") 反射获取 → 不触发 resolveClass()

3.4 为何用 HashSet 而非 BadAttributeValueExpException

Java 8u202 中，javax.management.BadAttributeValueExpException.readObject() 已被修复，不再调用 val 字段的 toString() 方法。因此改用 HashSet 作为触发入口（即 CommonsCollections6 链）：

map.put() → entry.hashCode() → TiedMapEntry.getValue() → LazyMap.get() → 链触发。

五、漏洞总结

5.1 漏洞清单

编号 漏洞类型 严重程度 说明
VULN-1 CWE-502: 不安全反序列化 严重 (Critical) POST / 端点直接对用户输入的字节流执行 readObject()
VULN-2 过滤绕过 高 (High) SafeObjectInputStream 黑名单仅有 6 个前缀，且仅检查序列化流中的类名，运行时反射调用完全不经过检查
VULN-3 已知漏洞依赖 高 (High) commons-collections:3.2.1 是历史上最著名的反序列化 gadget 库，公认存在严重安全隐患

5.2 防御措施

  1. 根本修复：完全禁止对不受信数据的反序列化。如需序列化，改用 JSON/Protobuf 等安全格式。
  2. 如果必须使用 Java 序列化：
    • 使用 ObjectInputFilter（Java 9+，Java 8u121+ 已 backport）设置全局过滤器，白名单只允许安全类
    • 不应使用黑名单，黑名单极易被绕过
  3. 升级 commons-collections：3.2.2+ 版本默认关闭了危险的 InvokerTransformer 功能，或在项目中根本不使用该库
  4. 移除 SecurityManager 假安全感：CustomSecurityManager 仅限制网络出口，不能防御 RCE，反而造成”已防护”的错觉
